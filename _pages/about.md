이를 기반으로 Spring Boot, 테스트 코드를 기반한 기능 구현 등과 같은 수강 내용을 **프로젝트에 적용해보며** 서비스를 구축한 경험이 있습니다.

> **담당 업무**
> * 광고 송출 서비스(서버)의 트래픽을 AWS CloudWatch 콘솔을 통해 모니터링, 대응
>   * 요청 트래픽 부하시 스케일링 그룹 AZ에 남은 리소스(하드웨어)가 부족한 경우 대응
>     * AMI로 다른 AZ의 인스턴스 생성 후 직접 로드밸런서에 연결
> * 그 외 AWS 인스턴스 관리, 비용 최적화
>   * 비용 절감 목적으로 SpotInst(베스핀 기업의 스팟인스턴스 관리 서비스) 사용 테스트
>     * 해당 서비스의 불안정한 상태 때문에 실제 적용은 하지 못함
>   * 비효율적인 인스턴스 유형을 사용하는 경우 변경

> **문제 해결 사례**
> * 특정 DB 인스턴스의 CPU 사용량이 불특정한 주기로 급증하는 상태 확인
> * 트래픽은 평소와 같아 슬로우 쿼리라고 판단, 슬로우 쿼리 로그에 기록된 쿼리를 모두 분석
>   * 클러스터 파라미터 그룹에서 slow_query_log 설정
> * 실행 계획이 풀 테이블 스캔인 슬로우 쿼리를 찾아 해당 테이블에 인덱스 생성하여 처리

> **배운 점**
> * slow_query_log 설정으로 슬로우 쿼리를 로그로 남길 시 딱 슬로우 쿼리만이 기록될 거라 예상
> * 하지만 쿼리의 실행 계획이 아닌 실제 쿼리 실행 시간을 기준으로 슬로우 쿼리가 판단됨
> * 따라서 동시에 실행되는 쿼리도 영향을 받아 실제 빠른 쿼리라도 로그에 남는 경우 발생
> * 슬로우 쿼리 기준 시간을 long_query_time으로 설정이 가능하나 기준 시간 판단이 어려움
> * 또한 쿼리의 실행 계획이 인덱스를 사용하지 않는 경우 log_queries_not_using_indexes로 확인 가능
