> **문제 해결 사례**
> * **DB CPU 사용량이 불특정한 주기로 급증**하는 상태 확인
>   * AWS CloudWatch 콘솔을 이용하여 모니터링
> * 하지만 해당 DB는 다양한 서비스에서 접근하는 DB로 **어떠한 모듈이 원인인지 확인하기 어려움**
>   * 개발팀 내에 공론화했지만, 팀원들 또한 알지 못함
> * 네트워크 트래픽 부하는 평소와 비슷함 (특별히 증가하지 않음)
> * 따라서 슬로우 쿼리가 있을 것이라 판단하고 슬로우 쿼리 존재 여부 확인
>   * RDS 클러스터 파라미터 그룹에서 Slow_query_log 옵션 설정으로 **슬로우 쿼리 남김**
>     * General_log 옵션 설정은 모든 쿼리를 로그로 남겨 성능이 현저히 느려져 사용하지 않음
>   * long_query_time 옵션값을 1로 슬로우 쿼리 시간 설정
>   * log_output 옵션값을 TABLE로 설정 (로그를 테이블에 저장)
>   * 문제는 성능 부하 타이밍에 같이 처리 중인 쿼리들이 로그에 모두 남음
>   * 결국 슬로우 쿼리 로그에 남은 모든 쿼리들의 **실행 계획 확인**
> * 찾아낸 조회 쿼리의 실행 계획이 풀 테이블 스캔인 것을 확인
>   * 해당 테이블은 정산 관련 데이터 테이블로 **수백만 건이 넘는 데이터가 저장**되어 있었음
> * 결론적으로 팀원이 기능을 추가하면서 작성한 쿼리로 인해 발생한 문제
> * 해당 기능에 대한 로직 수정이 어려워 특정 컬럼을 기준으로 **인덱스 생성**
>   * 쿼리의 실행 계획이 인덱스 레인지 스캔으로 변경되어 성능 향상
> * 이 외에도 CPU 사용량이 급증해 **해당 EC2 인스턴스 자체가 멈췄던 상황** 발생
>   * 팀원이 특정 모듈을 수정 후 배포한 직후에 발생
>   * 단순히 모듈의 성능 문제라고 볼 수 없다고 판단, 직접 로직 확인
>   * 특정 로직의 반복문이 무한 루프를 돈다는 것을 확인
>   * 원인은 다른 팀원의 특정 모듈의 잘못된 로직 수정으로 인하여 발생
>   * 모듈의 소스를 이전 커밋단계로 롤백하여 복구, 나중에 수정하여 재배포

> **담당 업무**
> * 고객사(매체)가 제공하는 RSS를 AMP 데이터로 변환하는 모듈 개발
>   * 매체의 RSS 데이터를 Jsoup를 사용, 크롤링 후 변환하여 저장기능 구현
>   * linux crontab으로 특정 시간마다 모듈을 실행
>   * Bitbucket(Web hook)과 Jenkins를 사용하여 배포 자동화 구현
>     * Bitbucket의 webhook 설정, 모듈과 같은 EC2 인스턴스의 있는 Jenkins와 연동
>     * Jenkins execute shell의 셸 스크립트를 작성하여 모듈의 리빌드, 실행 처리
> * AMP 페이지의 SEO(Search Engine Optimization)를 위해 구글 SEO 가이드 확인, 적용
>   * SEO의 명확한 기준 판단을 위해 구글 방문, 구글 개발자와 미팅 내용 문서화
>   * AMP 페이지가 구글 검색 상단에 노출, AMP 페이지의 게시된 광고를 통해 수익 발생
> * AMP 페이지 퍼블리싱 서버(PHP - CI) 유지보수, 운영
>   * AMP 페이지 유효성 확인
>     * [구글 AMP 유효성 검사 링크](https://search.google.com/test/amp) 활용
>     * 콘텐츠, 도메인 불일치 문제와 권장되는 URL 규격 준수 여부 확인
>     * site.map 파일, robots.txt 파일 유효성 확인
>     * 그 외 canonical 태그, 리디렉션 등 준수 사항 확인
>   * 매체별 규격에 맞춰 AMP(+AMP Story) 페이지 템플릿 구현, 수정
>   * Vue.js 상용 버전을 라이브러리 형태로 임포트하여 사용
>     * 디렉티브, 컴포넌트 등을 활용해 데이터 핸들링(렌더링)
>   * AMP 페이지에 광고 태그(amp-ad) 삽입

> **사용 기술**
> * Java, PHP
> * Spring Boot, MyBatis, Gradle, CI
> * AWS - EC2, RDS(Aurora)
> * Bitbucket, Jenkins

> **문제 해결 사례**
> * 해당 모듈과 같은 크롤링 유형의 모듈이 다수 존재한다는 것을 확인
>   * 또한 각기 다른 EC2 인스턴스에 산발적으로 배포되어 있어 관리가 어려움
>   * 문서와 문서 경로가 잘 관리되지 않아 문서를 찾기 어려움
>   * EC2 인스턴스 관리 차원에서도 모듈 관리 차원에서도 비효율적인 상태
> * 효율적인 인스턴스 관리를 위해 **모든 인스턴스에 태그 처리**하여 태그별로 확인
>   * 태그를 기준으로 모든 비슷한 유형 서비스의 소스 로직과 데이터 입출력, 트래픽 비교 분석
>   * 크롤링 모듈로 추정되는 모든 모듈의 로직 확인
> * 분석한 결과를 토대로 비슷한 유형의 수집 모듈 약 6개를 **하나의 모듈로 병합**
>   * 소스와 배포 자동화 등 관리 포인트 축소
> * 그 외 불필요하거나 더 사용되지 않는 EC2 **인스턴스 약 3개를 삭제**
>   * 월 기준 약 200달러 절약
> * 서비스의 목적과 개발환경, 대처 방안 등을 문서에 추가, 갱신 후 사내 공유

> **배운 점**

>   * 기존 모듈의 존재를 알고 활용하였다면 **생산성 측면**에서도 효율적이었을 것입니다.
> * 이런 환경은 장기적으로 관리 포인트가 늘어남을 뜻하면서 향후 유지보수, 운영에 있어서 많은 리소스가 들어간다는 것을 의미합니다.
>   * 또한, 이는 불필요한 AWS 인스턴스 사용으로 인해 낭비된 비용이 있음을 뜻합니다.
