### 트루닥(EMR) 서비스 운영 및 장애 대응, 인프라 관리
(2022.09 ~ 2022.11) 2개월

**업무**
* QueryDSL Qclass 초기화 시 Qclass 순환 참조로 발생한 데드락 장애 대응  
  서버 로딩 직후 QueryDSL을 사용하는 API를 2개 이상 동시 요청 시 발생  
  @Configuration 클래스 내 패키지 스캔 설정을 추가, 서버 로딩 시 바로 스캔, 싱글스레드에서 초기화 처리
* 컨테이너 메모리 사용량이 파드 메모리 설정보다 높아 발생한 Pod OOM Kill(137) 장애 대응  
  스프링부트 최대 메모리 설정이 약 `4GB`로 K8s 파드 설정(`2.5Gib`)보다 발생  
  파드의 limit 설정 변경(`4.5Gib`) 및 향후 장애 방지를 위해 request 값을 동일하게 변경
* `Xen citrix`를 사용해 온프레미스 서버의 스테이지 서버 구축 및 K8s 설정  
  VM 노드 각각 마스터 1개, 워커 3개 구축하며 메모리 스왑 비활성화 처리  
  kubeadm, kubelet, kubectl, ContainerD 등 설치, cgroup 설정 및 각 모듈 ArgoCD 배포 설정  
* ArgoCD를 통해 각 운영 서비스 릴리스 배포

**사용 기술**
* Kotlin(Spring Boot), JPA, K8s, Jenkins, ArgoCD, NCP, Github

**K8s 설치 시 주의사항**
* 최소 CPU `2 Core` 이상, 메모리 `2Gib` 이상일 것
* kubelet의 원활한 작동을 위해 반드시 메모리 스왑 비활성화(`sudo swapoff -a`)할 것  
  컨테이너 별 사용 측정이 어려운 것으로 보임
* 처음 K8s 설치 시 방화벽 등에 문제로 `IPtables --flush` (모든 정책 삭제) 적용
* K8s 설정 시 CNI(Container Network Interface)는 자동 설정 및 속도 등을 고려하여 flannel 사용
* 로딩된 서비스에 따라 노드마다 사용 메모리가 달라 `4Gib` 에서 `8Gib`로 변경

### AWS 비용 관리 및 리소스 상태 모니터링, 장애 대응
(2019.02 ~ 2019.03) 2개월

**업무**
* AWS RDS DB 서버의 인스턴스 스케일 업 (r4 > r5), S3 백업 기능 설정 추가
* EC2 스케일 아웃 시 AZ 내 가용 리소스가 부족한 상황 대응  
  Auto Scaling 그룹 설정의 다른 AZ 추가 및 VM 노드 직접 삽입하여 처리
* 슬로우 쿼리 확인 및 처리 (log_queries_not_using_indexes 옵션 활용)  
  slow_query_log 설정으로는 진짜 느리지 않은 쿼리까지 기록에 남아 위 옵션 활용
* 태그나 대시보드를 기준으로 사용되지 않는 서비스 모듈 및 AWS 리소스 정리  
  또는 사용하는 인스턴스 유형 변경
* 메가존 빌링 서비스를 활용해 월별 지출 비용 관리
* 비용 절감 목적으로 SpotInst(베스핀 기업의 스팟인스턴스 관리 서비스) 테스트 진행  
  서비스가 불안정하여 적용 실패

**사용 기술**
* EC2(AMI, ELB, Auto Scaling), RDS(Aurora), CloudWatch, CloudFront, IAM

**문제 해결**
* 트래픽이 평시와 비슷한 상태에서 DB CPU 사용량이 불특정한 주기로 급증하는 상태 확인
* AWS RDS 클러스터 파라미터 그룹의 slow_query_log 설정하면 슬로우 쿼리를 모두 볼 수 있음
  * 진짜 슬로우 쿼리가 아니더라도 해당 시점에 쿼리 실행 시간이 오래 소요되면 기록 됨
  * long_query_time으로 슬로우 쿼리 시간 설정 가능하나 시간만으로는 정확한 측정이 어려움
* 따라서 슬로우 쿼리 로그에 기록된 쿼리를 모두 실행 계획 확인
  * 실행 계획이 풀 테이블 스캔인 슬로우 쿼리를 찾아 해당 테이블에 인덱스 생성하여 처리
* 단순히 인덱스를 사용하지 않는 쿼리는 log_queries_not_using_indexes 옵션으로 확인 가능
